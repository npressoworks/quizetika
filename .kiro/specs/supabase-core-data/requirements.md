# Requirements Document - supabase-core-data

## Introduction
本仕様書は、Quizetika のデータ管理機構における Firebase Firestore 依存から Supabase (PostgreSQL) への移行に伴う、コアデータモデルの CRUD 操作、検索、およびトランザクション処理に関するユーザー視点の振る舞い要件を定義します。

## Boundary Context
- **In scope**:
  - ユーザープロフィール情報、フォロー/アンフォロー関係、および獲得バッジ情報の管理
  - クイズ情報、クイズに含まれる問題データの管理、およびクイズの検索・ジャンル別フィルタリング
  - ユーザーごとのクイズブックマーク管理
  - ユーザー向けアプリ内通知（通知数カウント含む）および管理者からのお知らせ管理
  - これらコアデータに関する単体テストの正常動作
- **Out of scope**:
  - クイズの解答履歴（アテンプト）、解答中のAI対話、レーティングやリアクションなどのゲームプレイ関連（`supabase-gameplay` が担当）
  - モデレーション操作、タグ/ジャンル統合の投票・可決処理、サブスクリプション権限判定などのガバナンス関連（`supabase-governance` が担当）
  - 画像やアイコンなどのバイナリファイルアップロードおよびストレージ操作（`supabase-storage-migration` が担当）
- **Adjacent expectations**:
  - 本スペックは `supabase-auth-migration` から提供される認証済みのユーザー識別子（UID）を使用してデータを紐付けます。

## Requirements

### 1. ユーザー管理要件
**Objective:** ユーザーとして、自分のプロフィールを更新し、他のユーザーをフォローし、条件を満たしたときにバッジを獲得できるようにしたい

#### Acceptance Criteria
1. When ユーザーがプロフィール情報を更新した時, the ユーザー管理システム shall 更新後のプロフィールデータを永続化する。
2. When ユーザーが他のユーザーをフォローした時, the ユーザー管理システム shall トランザクション内でフォロー関係を作成し、双方のフォロー/フォロワー数を整合性を保って更新する。
3. When ユーザーが他のユーザーのフォローを解除した時, the ユーザー管理システム shall トランザクション内でフォロー関係を削除し、双方のフォロー/フォロワー数を整合性を保って更新する。
4. When ユーザーがバッジ獲得条件を達成した時, the ユーザー管理システム shall 重複登録を防ぎつつ、獲得バッジデータをトランザクション内で追加する。

### 2. クイズ・問題管理および検索要件
**Objective:** クイズ作成者として、クイズや問題を新規作成・編集・削除し、プレイヤーがそれらを検索またはジャンル別に一覧取得できるようにしたい

#### Acceptance Criteria
1. When ユーザーがクイズを新規作成した時, the クイズ管理システム shall 入力情報をクイズデータとして登録し、作成したクイズIDを応答する。
2. When クイズ作成者がクイズ情報を更新した時, the クイズ管理システム shall 指定されたクイズのタイトルや設定を更新する。
3. When クイズ作成者がクイズ内の問題一覧を並び替えた時, the クイズ管理システム shall トランザクション内で各問題の表示順序を正確に更新する。
4. When ユーザーがキーワード検索を実行した時, the クイズ管理システム shall キーワードに部分一致するクイズ一覧を適合度順に返却する。
5. When ユーザーが特定のジャンルを指定してクイズ一覧を要求した時, the クイズ管理システム shall 指定ジャンルの正規化されたIDに紐づくクイズ一覧をページネーション形式で返却する。

### 3. ブックマーク管理要件
**Objective:** ユーザーとして、気に入ったクイズをブックマークに保存し、後から一覧で確認できるようにしたい

#### Acceptance Criteria
1. When ユーザーがクイズをブックマークに追加した時, the ブックマークシステム shall ユーザーIDとクイズIDのペアを登録する。
2. When ユーザーがブックマークを削除した時, the ブックマークシステム shall 対象のブックマーク登録を削除する。
3. When ユーザーが自分のブックマーク一覧を要求した時, the ブックマークシステム shall 登録日時が新しい順にブックマーク済みのクイズ一覧を返却する。

### 4. 通知・お知らせ管理要件
**Objective:** ユーザーとして、自分宛ての通知やお知らせをタイムリーに取得し、未読の通知数を確認できるようにしたい

#### Acceptance Criteria
1. When アプリ内で通知イベントが発生した時, the 通知システム shall 対象ユーザー宛ての通知レコードを作成する。
2. When ユーザーが未読通知一覧を要求した時, the 通知システム shall 未読フラグが立っている通知の一覧を返却する。
3. When ユーザーが通知詳細を閲覧するか既読操作を行った時, the 通知システム shall 対象の通知を既読状態に更新する。
4. When ユーザーが管理システムからのお知らせ一覧を要求した時, the お知らせシステム shall 公開開始日時が現在時刻以前であり、公開終了日時が未設定または未来であるお知らせ一覧を返却する。

### 5. 残存する直接 Firestore 依存の排除

**Objective:** 開発者として、ユーザー・通知・お知らせ・検索ログドメインの API Route と UI コンポーネントが Firestore へ直接依存する箇所を残さないようにしたい。それにより、`supabase-cleanup` が Firebase パッケージを安全に削除できるようにしたい。

#### Acceptance Criteria
1. The `src/app/api/user/delete-account/route.ts`, `src/app/notifications/announcements-tab.tsx`, `src/app/notifications/notifications-client.tsx`, `src/app/api/search/weekly-top/route.ts` は firebase および firebase-admin パッケージ、`@/lib/firebase/*` への import を持たない。
2. When アカウント削除リクエストを処理する時, the アカウント削除処理 shall Supabase サーバークライアントを用いて対象ユーザーの削除ステータスを更新する。
3. When 週間人気検索キーワード・タグの集計 API が呼び出された時, the 集計処理 shall Supabase の `search_logs` テーブルを参照する。
