# Requirements Document

## Introduction

本仕様書は、ローカル環境で `npm run test:e2e` を実行した際に発生する156件中50件のE2Eテスト失敗について、それぞれの根本原因を調査・分類し、Firebase削除・Supabase移行作業（`supabase-cleanup` スペック）とは独立して修正するための要件を定義します。対象の失敗は単一原因（`NEXT_PUBLIC_DISABLE_ADS` 設定等）では説明できず、バッジ付与・管理者ポータル・ソーシャル機能・リーダーボード・学習支援・ストリーミングスケルトン・クイズ検索・SEO共有など機能ドメイン全般に及ぶことが切り分け済みです。`supabase-cleanup` スペックのタスク5.3で発見・修正済みの2件の既存バグ（`saveQuiz` のINSERT順序バグ、`applyCursorFilter` のthenable自動実行バグ、コミット `d067070`）はこの母集合から除外されており、本スペックはその修正後もなお残存する失敗群を対象とします。

## Boundary Context

- **In scope**:
  - ローカル Supabase 環境（`supabase db reset` 後の状態）で `npm run test:e2e` を実行した際に検出される失敗テストの一覧化・原因調査・分類
  - 特定された根本原因（プロダクトコードの不具合、テストコード自体の不備、テスト環境・設定の問題）に応じた修正の実施
  - 修正内容を裏付けるリグレッションテスト（Jest単体テストまたはPlaywright E2Eテスト）の追加・更新
  - `npm run test` および `npm run test:e2e` による最終検証ゲート
- **Out of scope**:
  - Firebase関連コード・設定・パッケージの削除作業そのもの（`supabase-cleanup` スペックで完了済み）
  - `saveQuiz` のINSERT順序バグ、`applyCursorFilter` のthenable自動実行バグの再修正（コミット `d067070` で対応済み）
  - 新機能の追加やUI/UXの変更（バグ修正に必要な最小限の変更を除く）
  - Firebase削除・Supabase移行と無関係な既存機能へのパフォーマンス最適化やリファクタリング
- **Adjacent expectations**:
  - 本スペックの開始前提として、`supabase-cleanup` スペックのタスク5.3までの成果（コミット `d067070` 時点のコードベース）が適用済みであること
  - 本スペックの修正はサービス層（`src/services/*`）・コンポーネント・E2Eテストコードのいずれにも及ぶ可能性があるが、いずれの場合も対象ドメインの機能仕様（`.kiro/steering/product.md` 記載のコア機能）を変更しない

## Requirements

### 1. ベースラインの確立と再現性の確保

**Objective:** 調査担当者として、現在失敗しているE2Eテストの正確な一覧（ファイル名・テストタイトル・エラー内容）を再現可能な形で確定したい。それにより、以降の調査・修正・検証が一貫した母集合を対象にできるようにしたい。

#### Acceptance Criteria

1. When 調査プロセスが開始される時, the 調査プロセス shall ローカル Supabase を `supabase db reset` でリセットした状態で `npm run test:e2e` を実行し、失敗した各テストのファイル名・テストタイトル・エラーメッセージを記録する。
2. The 調査プロセス shall 記録した失敗テスト一覧を、本スペックが対応責任を負うベースラインとして確定する。
3. If 同一テストを複数回再実行した結果、成功と失敗が再現性なく入れ替わる場合, then 調査プロセス shall 当該テストを機能不具合起因の失敗とは区別し、flaky（不安定）である旨を記録する。
4. While ベースラインを取得する間, the 調査プロセス shall `supabase-cleanup` スペックの完了状態（Firebase関連コード削除済み）を前提とし、Firebase関連コードへの変更を行わない。

### 2. ドメイン別の根本原因調査

**Objective:** 調査担当者として、ベースラインの各失敗テストについて根本原因を特定し分類したい。それにより、対症療法ではなく実際の欠陥に対して修正を行えるようにしたい。

#### Acceptance Criteria

1. When ベースラインの失敗テストを調査する時, the 調査プロセス shall 失敗原因を「プロダクトコードの不具合」「テストコード自体の不備（セレクタ・待機処理・アサーション誤り等）」「テスト環境・設定の問題」のいずれかに分類する。
2. The 調査プロセス shall バッジ付与・管理者ポータル・ソーシャル機能・リーダーボード・学習支援・ストリーミングスケルトン・クイズ検索・SEO共有を含む、ベースラインで検出された全てのドメインについて根本原因調査を実施する。
3. If 複数の失敗テストが同一の根本原因を共有する場合, then 調査プロセス shall それらをグループ化し、単一の修正対象として扱う。
4. The 調査プロセス shall 特定した各根本原因について、再現手順と該当するソースファイル・行を記録として残す。
5. If 調査の結果、原因が `supabase-cleanup` スペックのスコープ（Firebase関連コード）に起因すると判明した場合, then 調査プロセス shall 当該項目を本スペックの対象から除外し、その理由を記録する。

### 3. 根本原因ごとの独立した修正

**Objective:** 開発者として、特定された根本原因それぞれに対して独立した最小限の修正を行いたい。それにより、Firebase削除作業や無関係な機能に影響を与えずに欠陥を解消したい。

#### Acceptance Criteria

1. When 根本原因に対する修正を実施する時, the 修正プロセス shall 当該原因に対応するプロダクトコードまたはテストコードのみを変更し、無関係なリファクタリングや機能追加を行わない。
2. The 修正プロセス shall Firebase削除・Supabase移行に関するコード（`supabase-cleanup` スペックの対象範囲）を変更対象に含めない。
3. If 修正対象がプロダクトコードの不具合である場合, then 修正プロセス shall 実際の機能不具合を修正し、不具合のある挙動に合わせてテストの期待値を変更してはならない。
4. If 修正対象がテストコード自体の不備である場合, then 修正プロセス shall プロダクトコードを変更せずテストコードのみを修正する。
5. While 個々の根本原因を修正する間, the 修正プロセス shall `.kiro/steering/product.md` に記載された対象ドメインの機能仕様を変更しない。

### 4. リグレッションの防止

**Objective:** 開発者として、修正した不具合が将来再発しないことを保証したい。それにより、同種の欠陥が再度見過ごされることを防ぎたい。

#### Acceptance Criteria

1. When プロダクトコードの不具合を修正した時, the 修正プロセス shall 当該不具合を再現する自動テスト（Jest単体テストまたはPlaywright E2Eテスト）を追加または更新する。
2. While 個々の修正を行う間, the 修正プロセス shall 修正前に成功していた既存のJestテストおよびPlaywrightテストを退行させない。
3. When 1件の根本原因に対する修正が完了した時, the 修正プロセス shall `npm run test` を再実行し、既存テストへの悪影響がないことを確認する。

### 5. 最終検証ゲート

**Objective:** 開発担当チームとして、ベースラインで特定された全ての失敗が解消され、既存テストへの悪影響がないことを最終確認したい。それにより、E2Eテストスイート全体の健全性を保証したい。

#### Acceptance Criteria

1. When Requirement 1で確定したベースラインの失敗全てに対する修正が完了した時, the 検証プロセス shall ローカル Supabase をリセットした状態で `npm run test:e2e` を再実行する。
2. The 検証プロセス shall 再実行結果においてベースラインの失敗テストが全て成功していることを確認する。
3. If Requirement 1.3でflakyと分類されたテストが再実行後も不安定な挙動を示す場合, then 検証プロセス shall その旨を記録した上で、機能不具合に起因しないことを条件として本要件の完了判定を妨げないものとする。
4. If 再実行の結果、ベースラインに含まれていなかった新規の失敗（デグレード）が検出された場合, then 検証プロセス shall 本スペックを完了状態として報告せず、当該デグレードを追加調査対象とする。
5. When `npm run test:e2e` の全件成功が確認された時, the 検証プロセス shall `npm run test` を実行し、単体テストスイート全体が成功することを最終確認する。
