# 要件定義書: quizetika-auth-profile-ui

## はじめに
本ドキュメントは、クイズ投稿SNS「quizetika」におけるユーザー認証、プロフィール管理、ソーシャル連携、通知、および本人向けプレイ履歴表示のフロントエンドUI要件を定義します。

**Phase 5（2026-06）**: ログイン中ユーザーが自身のプロフィールで、完了済みプレイ履歴を一覧・ページング表示し、クイズ詳細へ再遷移できるUIを追加する（データ取得APIは `quizetika-core` が担当）。

**Phase 8（2026-06）**: プロフィールの「作成したリスト」タブで、クイズリストと問題リスト（`listType`）をユーザーが区別できる表示に拡張する（`listType` の永続化と取得フィルタは `quizetika-core` が担当）。

**Phase 12（2026-06）**: 主要画面（ブックマーク、通知）の Server Component 化、静的フレーム先行表示、および Suspense と Streaming による非同期スケルトン表示への移行を追加します。

**Phase 23（2026-06-09）**: 廃止されたリアクション機能に伴い、本人プロフィールから「リアクション履歴」への導線を削除します（`/profile/[uid]/likes` ルート本体の即時削除は本フェーズの対象外）。

**Phase 26（2026-06-10）**: リスト機能完全廃止に伴い、プロフィール画面から「作成したリスト」タブおよび関連コンポーネント・テストを除去します（要件 11 参照）。要件 8（Phase 8 リストタイプ別表示）は本フェーズで **廃止** します。

**Phase 27（2026-06-23）**: プロフィール画面の「作成したクイズ」タブにおいて、クイズをページング（1ページあたり指定件数表示）し、かつキーワード（タイトル・説明・タグ・ジャンル）でリアルタイムに検索・フィルタリングできるUI要件を追加する。

**Phase 28（2026-06-27）**: プロフィール画面および編集画面において、ユーザーが好きなジャンルを設定・表示できるUI要件を追加する。

**Phase 29（2026-06-28）**: プロフィールメニュー表示名を「プロフィール」に変更し、PCサイドバーおよびモバイルヘッダーのドロップダウンメニューにダッシュボードリンクを追加する。また、他人のプロフィール画面において本人のプレイ履歴が露出しないよう、表示制御を厳格化する。

**Phase 30（2026-07-14）**: プロフィール編集画面からアバター画像を変更できるUI要件（要件16）、およびプロフィールコンテンツタブの視認性を向上させるUI要件（要件17）を追加する。

**Phase 31（2026-07-16）**: プロフィール編集画面のアバター画像変更操作に、円形のトリミング（クロップ）UIを追加する（要件18）。Phase 30 で対象外とされていたアバター画像のクロップ機能を、クイズカバー画像のトリミング機能（`quizetika-quiz-image-upload`）と同様のロジックを踏襲する形で対象内に変更する。

## 境界コンテキスト
- **対象範囲（In scope）**:
  - ソーシャルサインイン（Google / X / Microsoft）および開発用簡易ログインを含む認証画面とログイン状態に応じたリダイレクト。
  - プロフィール画面（`/profile/[uid]`）: 基本情報、バッジ、信頼スコア・権限ティアー、作成クイズタブ、フォロー操作、弱点克服導線（本人）、退会処理中フォールバック。
  - **Phase 5**: 本人プロフィールのコンテンツ領域に「プレイ履歴」**専用タブ** — クイズ名・スコア・モード・日時・経過時間・クイズ詳細への導線、追加読み込み。
  - プロフィール編集（`/profile/edit`）、フォロー／フォロワー一覧、通知一覧。
  - **Phase 23**: 本人プロフィールから「リアクション履歴」ボタン／リンクの削除。
  - **Phase 26**: プロフィール「作成したリスト」タブおよびリスト専用コンポーネント・取得ロジックの除去（要件 11）。
  - **Phase 12**: ブックマーク一覧（`/bookmarks`）および通知一覧（`/notifications`）の Server Component 化、静的フレーム（戻るボタン、タイトルヘッダー、基本コンテナなど）の先行レンダリング、データ解決待ち中の部分スケルトン表示（`Suspense` 適用）。
  - **Phase 27**: プロフィール画面の「作成したクイズ」タブ内における、クイズ一覧に対するキーワード検索入力（タイトル、説明、ジャンル、タグ対象）とフィルタ表示、および「前へ」「次へ」・ページ番号によるクライアントサイドのページングUI。
  - **Phase 28**: プロフィール編集画面における好きなジャンルの複数選択・保存UI、およびプロフィール画面における選択済みジャンルのアイコン・テキストチップ表示。
  - **Phase 29**: プロフィール関連メニュー（サイドバー等）の「プロフィール」から「プロフィール」への文言変更。
  - **Phase 29**: アバタークリック時のドロップダウンメニュー（PCサイドバーおよびモバイルヘッダー）への「ダッシュボード」リンクの追加。
  - **Phase 29**: 他人のプロフィール画面（`/profile/[uid]`）における、本人プレイ履歴の非表示ガードの強化（タブコンテンツのレンダリング抑止）。
  - **Phase 30**: プロフィール編集画面（`/profile/edit`）からのアバター画像の変更（選択・プレビュー・アップロード・保存）。
  - **Phase 30**: プロフィール画面のコンテンツタブ（「作成したクイズ」／「プレイ履歴」）の視認性向上（選択状態の明確化、タップ領域拡大等）。
  - **Phase 31**: アバター画像変更操作における、円形のトリミング（クロップ）UI（拡大縮小・位置調整・正方形範囲への制約・上限解像度への縮小・JPEG変換）の提供（要件18）。
- **対象外（Out of scope）**:
  - `attempts` の永続化、プレイ履歴クエリAPI、IDトークン検証（`quizetika-core`）。
  - 他ユーザーのプロフィールからのプレイ履歴閲覧（非公開）。
  - クイズプレイ・作成画面UI（`quizetika-play-flow-ui` / `quizetika-creator-dash-ui`）。
  - 退会時の Firestore 匿名化クレンジング本体（`quizetika-core`）。
  - **Phase 8（リスト関連は Phase 26 で廃止）**: リストの作成・編集・`listType` 選択 UI（`quizetika-creator-dash-ui`）。ブックマーク3タブ・問題リスト連続プレイ（`quizetika-play-flow-ui`）。
  - **Phase 12**: プロフィール画面（`/profile/[uid]`）やフォロー一覧、リアクション履歴などの非同期表示最適化（主要画面以外は本フェーズの対象外）。
  - **Phase 23**: `/profile/[uid]/likes` ルートの削除または404化、リアクション機能データ（`reactions` コレクション）のマイグレーション、リアクション関連 E2E の一括削除（導線削除と本人プロフィール UI 更新に限定する）。
  - **Phase 26**: リストデータの Firestore 削除・マイグレーション（`quizetika-core`）。`/list/*` ルート・ブックマークリストタブ除去（`quizetika-play-flow-ui`）。Sidebar「リスト」ナビ（`quizetika-sidebar-layout`）。
  - **Phase 30（Phase 31 で一部変更）**: アバター画像の回転・フィルタ等、円形トリミング以外の画像編集機能（要件18の対象範囲外）。差し替え前アバター画像の Storage 上の物理削除・クレンジング（`quizetika-account-deletion-cleansing` 等が担当）。
  - **Phase 31**: トリミングUIコンポーネント自体をクイズカバー画像機能と共有・共通化するかどうかの実装方針（設計フェーズで決定）。円形以外のアスペクト比・トリミング形状の提供。
- **隣接システムへの期待**:
  - プレイ履歴は `GET /api/user/play-history`（`Authorization: Bearer`）で取得し、トークン本人の `uid` のみ返る。クエリで他 `uid` を指定しても 403。
  - 初回20件 + `nextCursor` による追加ページ。`test-play` はAPI側で除外済み。
  - **Phase 26**: リスト関連 API（`getQuizListsByAuthor` 等）は存在しない前提。プロフィールは「作成したクイズ」タブと本人のみ「プレイ履歴」タブを表示する。
  - **Phase 12**: ログイン必須画面に対する Next.js Middleware でのサーバーサイドリダイレクト（Cookie ベース認証）。
  - **Phase 23**: Sidebar アカウントポップアップからの設定導線は `quizetika-sidebar-layout` が担当すること。設定ページ本体は `quizetika-user-settings-ui` が担当すること。

## 要件

### 要件 1: ユーザー認証画面 (`/login`)
**目的:** ゲストユーザーとして、ソーシャルアカウントで安全にログインしたい。それにより quizetika の機能を利用できる。

#### 受け入れ基準
1. 認証画面は Google・X（Twitter）・Microsoft（Azure AD）のサインインボタンを表示し、Firebase Auth のポップアップで認証を開始すること。
2. 認証に成功したとき、認証画面は `redirect` クエリがあればその遷移先へ、なければホーム（`/`）へ遷移すること。
3. 既にログイン済みのユーザーが認証画面（`/login`）に直接アクセスした場合、ホーム（`/`）へ自動リダイレクトすること。
4. ポップアップブロック・キャンセル・プロバイダ未有効化等の Firebase エラーは、日本語の説明メッセージとして画面に表示すること。
5. 開発・E2Eテスト環境（`NODE_ENV !== 'production'` かつ `NEXT_PUBLIC_ENV === 'test'`）ではのみ、メール／パスワードによる簡易ログインを表示してよい（本番では表示しない）。

### 要件 2: プロフィール画面 (`/profile/[uid]`)
**目的:** Quizetikaのユーザーとして、他者または自身のプロフィールを確認し、つながりや投稿内容を把握したい。

#### 受け入れ基準
1. プロフィール画面は、対象ユーザーのアバター、表示名、自己紹介、信頼スコア、権限ティアーバッジ、獲得称号バッジ一覧を表示すること。
2. プロフィール画面は「作成したクイズ」タブパネルを表示すること。本人プロフィールの場合は第2タブとして「プレイ履歴」を追加すること（要件 7）。**Phase 26 以降**、「作成したリスト」タブを表示してはならない（要件 11）。
3. ログイン中ユーザーが他ユーザーのプロフィールを閲覧している場合、フォロー／フォロー解除トグルを表示し、操作時に `UserService` 経由で接続状態とフォロワー数表示を更新すること。
4. 対象プロフィールの `deleteStatus` が `delete_pending` のとき、本人以外のアクセスはブロックし、404相当の表示とすること（本人には削除処理中の警告を表示してよい）。
5. 本人プロフィールで `totalFailedQuestionsCount > 0` のとき、弱点克服（復習）セクションと `/quiz/review` への導線を表示すること。
6. 本人プロフィールをログイン中ユーザーが閲覧している場合、プレイ履歴の表示領域を表示すること（詳細は要件 7）。
7. **Phase 23 以降**: 本人プロフィールのアクション領域に「リアクション履歴」ボタンまたは `/profile/[uid]/likes` へのリンクを表示してはならない（要件 10）。

### 要件 3: プロフィール編集画面 (`/profile/edit`)
**目的:** 認証ユーザーとして、公開プロフィール情報を更新したい。

#### 受け入れ基準
1. プロフィール編集画面は、表示名と自己紹介の編集フォームを表示すること。
2. 表示名は最大30文字、自己紹介は最大200文字の入力制限を適用すること。
3. 「保存する」を押し入力が有効なとき、`UserService.updateProfile` を呼び出し、成功後に自身のプロフィール画面へ遷移すること。
4. **Phase 30 以降**: プロフィール編集画面は、アバター画像の変更操作を表示すること（詳細は要件16）。

### 要件 4: フォロー／フォロワー一覧画面 (`/profile/[uid]/connections`)
**目的:** Quizetikaのユーザーとして、つながりを確認しソーシャルネットワークを管理したい。

#### 受け入れ基準
1. 接続一覧画面は「フォロー中」「フォロワー」をタブで切り替え表示すること。
2. 各ユーザーカードはアバター、表示名、自己紹介を表示し、ログイン中ユーザー自身以外にはフォロー／フォロー解除トグルを表示すること。

### 要件 5: 通知一覧画面 (`/notifications`)
**目的:** Quizetikaのユーザーとして、アクティビティ通知を時系列で確認したい。

#### 受け入れ基準
1. 通知一覧画面は、フォロー通知やクイズ指摘修正完了などのアクティビティを時系列で表示すること。
2. 指摘修正完了系の通知カードをクリックしたとき、該当クイズの詳細画面（`/quiz/[id]`）へ遷移すること。

### 要件 6: リアクション履歴画面 (`/profile/[uid]/likes`)（レガシー・Phase 23 以降は導線廃止）
**目的:** （レガシー）Quizetikaのユーザーとして、送受信したリアクション履歴を確認したい。

> **Phase 23 注記**: リアクション機能は廃止方向のため、**プロフィールからの導線は要件 10 で削除**する。本要件はルート存続時のレガシー参照とし、新規導線の追加や画面改修は行わない。

#### 受け入れ基準（レガシー・メンテナンス対象外）
1. The [Auth Profile UI] shall [Phase 23 以降、プロフィール画面からリアクション履歴画面への新規導線を追加しないこと（要件 10 が優先）]。
2. Where [`/profile/[uid]/likes` ルートが存続している場合], the [Auth Profile UI] may [既存実装どおりタブ付き履歴一覧を表示し続けてよいが、本フェーズでの改修は必須としない]。

### 要件 7: 本人プレイ履歴表示（Phase 5）
**目的:** ログインユーザーとして、過去に完了したクイズプレイを自分のプロフィールで振り返し、再度プレイしたい。それにより学習・復習の導線を得られる。

#### 受け入れ基準
1. ログイン中ユーザーが**自身の**プロフィールを閲覧しているときのみ、コンテンツタブに「プレイ履歴」を表示すること。他ユーザーのプロフィールでは当該タブを表示してはならない。
2. ユーザーが「プレイ履歴」タブを選択したとき（初回選択時）、`GET /api/user/play-history` を Firebase ID トークン（Bearer）付きで呼び出し、完了日時の降順でエントリを表示すること。
3. 各履歴行は、クイズタイトル（クイズ詳細へのリンク）、正解数／総問題数、プレイモードの表示ラベル、完了日時、合計解答時間（秒）を含むこと。
4. 履歴が0件のとき、「まだプレイ履歴がありません」等の空状態を表示すること。
5. APIレスポンスに `nextCursor` があるとき、「もっと見る」等の操作で `cursor` クエリを付与して追加取得し、既存リストに追記すること（初回取得件数はAPIデフォルト20件に合わせる）。
6. 取得エラー（401／403／500）時は、再試行またはログイン案内を含むユーザー向けメッセージを表示し、クラッシュさせないこと。
7. E2E用に、プレイ履歴領域に `data-testid="play-history-section"`、各行に `data-testid="play-history-entry"`、追加読み込みボタンに `data-testid="play-history-load-more"` を付与すること。
8. プレイ履歴の永続化・除外ルール（`test-play` 除外等）をフロントエンドで再実装してはならない（`quizetika-core` のAPI結果を信頼する）。

### 要件 8: 作成リストのタイプ別表示（Phase 8）— **Phase 26 で全体廃止**
**目的:** （廃止）プロフィール閲覧者として、ユーザーが作成したクイズリストと問題リストを区別して把握したい。それによりリストの内容種別を誤解なく選べる。

> **Phase 26 注記**: リスト機能は完全廃止のため、本要件は **履歴参照のみ**。プロフィールからのリストタブ除去は **要件 11 が正本**。

#### 受け入れ基準
1. ユーザーがプロフィールの「作成したリスト」タブを表示したとき、プロフィール UI は、対象ユーザーが作成したすべてのリスト（公開・非公開をプロフィール所有者の既存ルールに従う）をカード一覧で表示すること。
2. リストカードを描画するとき、プロフィール UI は、`listType` が `question` のリストに「問題リスト」、`quiz` または未設定のリストに「クイズリスト」の種別ラベルを表示すること。
3. クイズリストのカードを描画するとき、プロフィール UI は、収録クイズ数（`quizIds` の件数）を表示すること。
4. 問題リストのカードを描画するとき、プロフィール UI は、収録問題数（`questionIds` の件数）を表示し、`quizIds` 件数を問題リストの収録数として表示してはならない。
5. リストカードをクリックしたとき、プロフィール UI は、リスト詳細画面（`/list/[id]`）へ遷移すること。
6. 本人プロフィールでリストが0件のとき、プロフィール UI は、既存と同様の空状態と「新しいリストを作成する」導線（`/list/create`）を表示すること。
7. 本人プロフィールのリストタブにフィルタ UI を含める場合、プロフィール UI は、「すべて」「クイズリストのみ」「問題リストのみ」から一覧を絞り込めること（初版は種別ラベルのみでも可。フィルタは任意実装）。
8. プロフィール UI は、`listType` の付与やリスト CRUD を実装してはならない（表示と遷移のみ。データは `quizetika-core` のリスト取得 API に依存する）。
9. E2E用に、リストカードに `data-testid="profile-list-card"`、種別ラベルに `data-testid="profile-list-type-badge"` を付与すること。

### 要件 9: ブックマーク・通知画面の非同期表示最適化（Phase 12 追加）
**目的:** 認証ユーザーとして、ブックマークや通知一覧にアクセスした際、画面全体の白紙ローディングを待つことなく、静的なタイトルやタブ切り替え、戻るボタンが即座に表示され、データが揃った箇所から順番にブックマーク一覧や通知リストが表示されるようにしたい。これにより、待機時のストレスや画面の点滅による不快感を防ぎ、スムーズに確認を行える。

#### 受け入れ基準

**ブックマーク一覧画面における非同期表示最適化**
1. When [ユーザーがブックマーク一覧画面（`/bookmarks`）にアクセスしたとき], the [Auth Profile UI] shall [サーバーコンポーネントとして静的な戻るボタン、タイトル、およびタブのヘッダーを含む静的なページフレームを即座にレンダリングし、Next.js of Streaming機能を通じてクライアントへ送信すること]。
2. While [ブックマークデータ（クイズ、リスト、問題）の読み込み中である間], the [Auth Profile UI] shall [コンテンツ表示エリアに、ブックマーク専用のスケルトンプレースホルダーを表示すること]。
3. When [ブックマークデータのロードが完了したとき], the [Auth Profile UI] shall [スケルトン表示領域を、実際のブックマーク一覧コンテンツ（タブに応じたクイズカード等）に差し替えること]。

**通知一覧画面における非同期表示最適化**
4. When [ユーザーが通知一覧画面（`/notifications`）にアクセスしたとき], the [Auth Profile UI] shall [サーバーコンポーネントとして静的な戻るボタン、タイトル、および「すべて既読にする」ボタン等のヘッダー要素を含む静的なページフレームを即座にレンダリングし、Next.js of Streaming機能を通じてクライアントへ送信すること]。
5. While [通知データの読み込み中である間], the [Auth Profile UI] shall [通知リスト表示エリアに、専用のローディングスケルトンプレースホルダーを表示すること]。
6. When [通知データのロードが完了したとき], the [Auth Profile UI] shall [スケルトン表示領域を、実際の通知カードリストに差し替えること]。

**アクセシビリティ・テスト支援**
7. The [Auth Profile UI] shall [ブックマーク一覧のスケルトン領域に `data-testid="bookmarks-skeleton"` を付与すること]。
8. The [Auth Profile UI] shall [通知一覧のスケルトン領域に `data-testid="notifications-skeleton"` を付与すること]。

**ログイン画面、プロフィール関連画面の非同期表示最適化**
9. When [ユーザーがプロフィール詳細画面（`/profile/[uid]`）または編集画面（`/profile/edit`）にアクセスしたとき], the [Auth Profile UI] shall [サーバーコンポーネントとして戻るボタン、アバター・ヘッダー枠、タブパネル等の静的フレームを即座にレンダリングし、Next.jsのStreaming機能を通じてクライアントへ送信すること]。
10. While [ユーザーのプロフィールデータ（アバター画像、フォロー数、バッジ情報等）、作成したクイズ・リスト、または本人のプレイ履歴リストがロード中である間], the [Auth Profile UI] shall [コンテンツエリアや各タブパネル内部に専用のスケルトンプレースホルダーを表示すること]。
11. When [データのロードが完了したとき], the [Auth Profile UI] shall [スケルトン表示領域を、実際のプロフィールコンテンツに差し替えること]。
12. When [ユーザーがフォロー／フォロワー一覧画面（`/profile/[uid]/connections`）、リアクション履歴画面（`/profile/[uid]/likes`）、またはログイン画面（`/login`）にアクセスしたとき], the [Auth Profile UI] shall [サーバーコンポーネントとしてヘッダー、戻るボタン等の静的フレームを即座にレンダリングし、クライアントへ送信すること]。
13. While [リストデータ（フォロー中ユーザー、いいねしたクイズ等）がロード中である間], the [Auth Profile UI] shall [リスト領域に専用のスケルトンを表示すること]。

**アクセシビリティ・テスト支援（追加分）**
14. The [Auth Profile UI] shall [プロフィール画面のスケルトン領域に `data-testid="profile-skeleton"`、フォロー一覧のスケルトン領域に `data-testid="connections-skeleton"` を付与すること]。

### 要件 10: リアクション履歴導線の削除（Phase 23）
**目的:** 認証ユーザーとして、廃止されたリアクション機能への迷い込みがないプロフィール体験を得たい。それにより現行機能と一致したプロフィール操作ができる。

#### 受け入れ基準

**本人プロフィールからの導線削除**
1. When [ログイン中ユーザーが自身のプロフィール（`/profile/[uid]`、`uid` が本人）を閲覧したとき], the [Auth Profile UI] shall [プロフィールアクション領域に「リアクション履歴」ボタン、リンク、または同等の導線を表示しないこと]。
2. The [Auth Profile UI] shall [本人プロフィールの主要アクションとして「プロフィールの編集」のみ、または編集に加えて現行で有効な導線（弱点克服等）を表示すること。リアクション履歴は含めないこと]。
3. When [他ユーザーのプロフィールを閲覧したとき], the [Auth Profile UI] shall [リアクション履歴への導線を表示しないこと（従来どおりフォロー操作等のみ）]。

**ルート存続とレガシー**
4. The [Auth Profile UI] shall [本フェーズでは `/profile/[uid]/likes` ルートファイルの削除または404化を必須としない（直接 URL アクセスは follow-up で整理可）]。
5. The [Auth Profile UI] shall [リアクション送信 UI、クリエイターリアクション、Firestore `reactions` データの削除を本要件の範囲に含めない]。

**テスト・隣接**
6. The [Auth Profile UI] shall [本人プロフィール画面に `data-testid="profile-reaction-history-link"` 等のリアクション履歴導線用 testid を付与しないこと]。
7. When [E2E テストが本人プロフィールのリアクション履歴導線の存在を検証している場合], the [Auth Profile UI] shall [当該テストの更新またはスキップを Phase 23 の直接実装候補（`remove-reaction-history-e2e`）と整合させること]。
8. The [Auth Profile UI] shall [設定画面（`/settings`）および Sidebar ポップアップ「設定」導線の追加を本要件の範囲に含めない（`quizetika-user-settings-ui` / `quizetika-sidebar-layout` が担当）]。

### 要件 11: プロフィールリストタブの除去（Phase 26）
**目的:** プロフィール閲覧者として、廃止されたリスト機能への導線がないプロフィール体験を得たい。それにより現行機能（作成クイズ・プレイ履歴）に集中できる。

#### 受け入れ基準

**タブ・表示の除去**
1. When [ユーザーがプロフィール画面（`/profile/[uid]`）を表示したとき], the [Auth Profile UI] shall [「作成したリスト」タブ、タブトリガー、または同等のリスト一覧パネルを表示しないこと]。
2. The [Auth Profile UI] shall [プロフィールコンテンツタブを「作成したクイズ」と、本人閲覧時のみ「プレイ履歴」の2タブ（または1タブ＋本人のみ第2タブ）構成とすること]。
3. The [Auth Profile UI] shall [`getQuizListsByAuthor` および `@/services/quiz-list` からのリスト取得呼び出しを実装してはならない]。

**コンポーネント・ライブラリの除去**
4. The [Auth Profile UI] shall [`ProfileListsPanel`、`ProfileListCard`、`profile-list-display` およびリスト専用 `data-testid`（`profile-list-card` 等）を提供してはならない]。
5. When [リスト専用コンポーネントファイルがリポジトリに残存している場合], the [Auth Profile UI] shall [当該ファイルを削除すること]。

**既存タブの維持**
6. The [Auth Profile UI] shall [「作成したクイズ」タブの一覧・空状態・クイズ詳細への導線を維持すること]。
7. When [ログイン中ユーザーが自身のプロフィールを閲覧したとき], the [Auth Profile UI] shall [要件 7 に従い「プレイ履歴」タブと `ProfilePlayHistoryPanel` を維持すること]。

**境界・隣接**
8. The [Auth Profile UI] shall [`/list/*` ルート削除・ブックマークリストタブ除去を本要件の範囲に含めない（`quizetika-play-flow-ui` が担当）]。
9. The [Auth Profile UI] shall [リストデータの Firestore 削除を本要件の範囲に含めない（`quizetika-core` マイグレーションが担当）]。

**アクセシビリティ・テスト支援**
10. The [Auth Profile UI] shall [リスト専用単体テスト（`profile-list-display`、`ProfileListCard`、`ProfileListsPanel`）およびプロフィールリストタブ前提の E2E を削除または非リストフローへ更新すること]。

### 要件 12: 作成したクイズのページングと検索機能（Phase 27）
**目的:** プロフィール画面の閲覧ユーザーとして、作成したクイズの一覧から特定のクイズを検索して探しやすくし、また多数のクイズがある場合でもページングによって読みやすくしたい。

#### 受け入れ基準
1. When [ユーザーがプロフィール画面の「作成したクイズ」タブを表示したとき], the [Auth Profile UI] shall [クイズ一覧の上部にキーワード検索用のテキスト入力欄と、下部にページングUI（ページ番号、前へ・次へボタン）を表示すること]。
2. When [ユーザーが検索入力欄にキーワードを入力したとき], the [Auth Profile UI] shall [クイズのタイトル、説明、ジャンル、またはタグのいずれかにそのキーワードが含まれるクイズのみに一覧をフィルタリングし、自動的に1ページ目にリセットすること]。
3. When [検索入力欄が空のとき], the [Auth Profile UI] shall [すべての作成したクイズをフィルタリングなしで表示すること]。
4. The [Auth Profile UI] shall [作成したクイズの一覧を1ページあたり9件で分割して表示すること（全体の件数が9件以下の場合は、ページングUIを非表示にするか、または非活性状態のページングUIを表示すること）]。
5. When [ユーザーがページングUIの「次へ」ボタンまたはページ番号をクリックしたとき], the [Auth Profile UI] shall [該当するページのクイズ一覧を表示し、表示エリアの最上部（またはクイズタブのコンテンツ開始位置）へスムーズにスクロールすること]。
6. When [ユーザーが1ページ目を表示しているとき], the [Auth Profile UI] shall [ページングUIの「前へ」ボタンを非活性化（disabled）すること]。
7. When [ユーザーが最終ページを表示しているとき], the [Auth Profile UI] shall [ページングUIの「次へ」ボタンを非活性化（disabled）すること]。
8. The [Auth Profile UI] shall [検索入力欄に `data-testid="profile-quiz-search-input"`、ページングUIに `data-testid="profile-quiz-pagination"` を付与すること]。

### 要件 13: 共通クイズカードの適用（Phase 27 拡張）
**目的:** プロフィール画面の閲覧ユーザーとして、ホームや検索画面と同一の操作性（ブックマーク切り替えやデザイン）でクイズカードを閲覧・操作したい。

#### 受け入れ基準
1. The [Auth Profile UI] shall [プロフィール画面の「作成したクイズ」タブ内のクイズカードについて、独自の実装を廃止し、ホーム・検索画面と共通の `QuizCard` コンポーネント（`@/components/quiz/quiz-card`）に置き換えて表示すること]。
2. The [Auth Profile UI] shall [ログイン中ユーザーのブックマーク情報（`bookmarkedIds`）を解決し、表示する各クイズのブックマーク状態（`isBookmarked`）を `QuizCard` に反映すること]。
3. When [ユーザーがクイズカード of ブックマークトグルボタンをクリックしたとき], the [Auth Profile UI] shall [ログイン済みの場合は `@/services/bookmark` の `toggleBookmark` を呼び出してブックマーク状態をトグルし、未ログインの場合はログイン画面（`/login`）へリダイレクトすること]。
4. When [ユーザーがクイズカードの「挑戦する」ボタンまたはカード自体をクリックしたとき], the [Auth Profile UI] shall [クイズ詳細画面（`/quiz/[id]`）へ遷移すること]。
5. The [Auth Profile UI] shall [共通 `QuizCard` の契約に従い、各クイズカードに `data-testid="quiz-card"` が付与されていることを確認すること（既存の `profile-quiz-card` は廃止または共通 testid に統合）]。

### 要件 14: 好きなジャンルの設定と表示（Phase 28）
**目的:** ユーザーとして、自分の好きなクイズのジャンルをプロフィールに設定し、また他ユーザーや自分のプロフィール画面でそれらを表示できるようにしたい。これにより、ユーザーの興味や専門分野が一目でわかるようになる。

#### 受け入れ基準
1. When [ユーザーがプロフィール編集画面（`/profile/edit`）を表示したとき], the [Auth Profile UI] shall [有効なジャンルの一覧をマスタから取得し、ユーザーが好きなジャンルを複数選択できるUIを表示すること]。
2. While [プロフィール編集画面でロードが完了したとき], the [Auth Profile UI] shall [ユーザーが既に登録している好きなジャンルを選択状態にして表示すること]。
3. When [ユーザーがプロフィール編集画面でジャンルを選択して保存を実行したとき], the [Auth Profile UI] shall [選択されたジャンルを保存し、完了後に自身のプロフィール画面へ遷移すること]。
4. When [ユーザーがプロフィール画面（`/profile/[uid]`）を表示したとき], the [Auth Profile UI] shall [対象ユーザーが設定した好きなジャンルの表示名とアイコン画像を含むチップの一覧を、プロフィール基本情報領域（自己紹介の周辺）に表示すること]。
5. When [対象ユーザーが好きなジャンルを1つも設定していないとき] かつ [ログインユーザー自身のプロフィールを表示しているとき], the [Auth Profile UI] shall [プロフィール画面に「好きなジャンルを登録しましょう」等の登録を促す控えめなテキストまたは導線を表示すること]。
6. When [対象ユーザーが好きなジャンルを1つも設定していないとき] かつ [他ユーザーのプロフィールを表示しているとき], the [Auth Profile UI] shall [好きなジャンルの表示領域全体を非表示にすること]。
7. The [Auth Profile UI] shall [プロフィール編集画面のジャンル選択UIに `data-testid="profile-genre-select"`、プロフィール表示画面の好きなジャンル表示領域に `data-testid="profile-favorite-genres"` を付与すること]。

### 要件 15: プロフィール表記変更、ダッシュボードリンク追加、およびプレイ履歴表示ガード（Phase 29）
**目的:** ユーザーとして、プロフィール関連機能へのアクセスを「プロフィール」という直感的な表現で利用でき、ダッシュボードへ簡単に遷移したい。また、他人のプロフィール画面で自分のプレイ履歴が表示されてしまうのを防ぎたい。

#### 受け入れ基準
1. When [ログイン中ユーザーがサイドバーなどのナビゲーションメニューを表示したとき], the [Auth Profile UI] shall [該当するメニュー項目名を「プロフィール」と表記すること（従来の「プロフィール」から変更）].
2. When [ログイン中ユーザーがアバターアイコンをクリックしてドロップダウンメニューを展開したとき], the [Auth Profile UI] shall [メニュー内に「ダッシュボード」への遷移リンクを表示すること].
3. When [ログイン中ユーザーが他人のプロフィール画面を表示したとき], the [Auth Profile UI] shall [プレイ履歴を表示するコンテンツエリアを決してレンダリングしないこと].
4. When [ログイン中ユーザーが自分自身のプロフィール画面（プロフィール）を表示したとき] かつ [プレイ履歴タブを選択したとき], the [Auth Profile UI] shall [プレイ履歴コンテンツエリアを表示すること].
5. The [Auth Profile UI] shall [アバターのドロップダウンメニュー内に表示するダッシュボードリンクに、それぞれ適切なテスト用識別子（`data-testid`）を付与すること].

### 要件 16: プロフィール編集画面からのアバター画像変更（Phase 30）
**目的:** 認証ユーザーとして、プロフィール編集画面から自分のアバター画像を変更したい。それにより自分らしさを表現でき、他ユーザーに視覚的に認識されやすくなる。

#### 受け入れ基準

**表示・選択**
1. When [ユーザーがプロフィール編集画面（`/profile/edit`）を表示したとき], the [Auth Profile UI] shall [現在設定されているアバター画像（未設定の場合はデフォルトアバター）と、画像を変更するための操作（ボタンまたはクリック可能なアバター領域）を表示すること]。
2. When [ユーザーが画像変更操作をトリガーし、許可された形式（PNG・JPEG・GIF）かつ既定の上限サイズ以下の画像ファイルを選択したとき], the [Auth Profile UI] shall [要件18のトリミングUIを表示し、ユーザーがトリミングを確定した画像をアバター領域にプレビュー表示すること（保存前は既存のアバター画像を変更しないこと）]。

**検証・エラー**
3. If [ユーザーが許可されていない形式のファイルを選択したとき], the [Auth Profile UI] shall [保存を行わず、許可形式を明示した日本語のエラーメッセージを表示すること]。
4. If [ユーザーが既定の上限サイズを超えるファイルを選択したとき], the [Auth Profile UI] shall [保存を行わず、上限サイズを明示した日本語のエラーメッセージを表示すること]。

**保存・更新**
5. When [ユーザーがプレビュー確認後にプロフィールの保存を実行したとき], the [Auth Profile UI] shall [選択された画像をアップロードし、アップロード完了後にユーザーのアバターURLを更新した上で、自身のプロフィール画面へ遷移すること]。
6. While [選択された画像のアップロード処理が進行中である間], the [Auth Profile UI] shall [保存操作を無効化し、処理が進行中であることを示す表示を行うこと]。
7. If [画像のアップロードまたは保存に失敗したとき], the [Auth Profile UI] shall [変更前のアバター表示を維持したまま、日本語のエラーメッセージを表示すること]。
8. When [ユーザーが画像を変更せずに表示名・自己紹介等の他項目のみを保存したとき], the [Auth Profile UI] shall [既存のアバター画像を変更しないこと]。

**反映**
9. When [アバター画像の変更保存が完了したとき], the [Auth Profile UI] shall [遷移先のプロフィール画面および既存のアバター表示箇所（ヘッダー、ドロップダウンメニュー等）に更新後の画像を反映すること]。

**アクセシビリティ・テスト支援**
10. The [Auth Profile UI] shall [アバター画像変更操作に `data-testid="profile-avatar-upload-input"`、プレビュー領域に `data-testid="profile-avatar-preview"` を付与すること]。

### 要件 17: プロフィールコンテンツタブの視認性向上（Phase 30）
**目的:** プロフィール閲覧者として、プロフィールのコンテンツタブ（「作成したクイズ」「プレイ履歴」）のうちどちらが選択中か、また各タブに何が表示されるかを一目で判別したい。それにより迷わず目的のコンテンツにたどり着ける。

#### 受け入れ基準
1. The [Auth Profile UI] shall [各コンテンツタブにアイコンとラベルの両方を表示し、選択中のタブを非選択のタブと視覚的に明確に区別できる配色・背景・下線等のスタイルで表示すること]。
2. When [ユーザーがタブを切り替えたとき], the [Auth Profile UI] shall [選択状態の視覚的な変化を即座に反映すること]。
3. The [Auth Profile UI] shall [モバイル幅の画面において、各タブが指で押しやすい十分な大きさのタップ領域を持つこと]。
4. The [Auth Profile UI] shall [「作成したクイズ」タブのラベルに、要件2で定義済みの表示件数（例:「作成したクイズ (n)」）を維持すること]。
5. The [Auth Profile UI] shall [キーボード操作またはスクリーンリーダー利用時に、現在選択中のタブを判別可能にすること]。
6. The [Auth Profile UI] shall [要件11で定義したタブ構成（「作成したクイズ」と、本人閲覧時のみ「プレイ履歴」）を変更しないこと（視覚的なスタイル改善に限定し、タブの追加・削除・並び替えを本要件の範囲に含めない）]。

### 要件 18: アバター画像の円形トリミング機能（Phase 31）
**目的:** 認証ユーザーとして、アップロードするアバター画像を円形の枠に合わせて直感的にトリミングしたい。それにより、意図した構図で見切れのない円形アバターをプロフィール上に表示できる。

#### 受け入れ基準

**表示・操作**
1. When [ユーザーがプロフィール編集画面で許可形式・上限サイズ以下の画像ファイルを選択したとき], the [Auth Profile UI] shall [選択画像を表示したトリミングモーダルを開き、円形のトリミング枠、拡大・縮小操作、および位置調整操作を提供すること]。
2. While [トリミングモーダルが表示されている間], the [Auth Profile UI] shall [トリミング範囲を常に正方形（1:1）の領域に制約し、円形の枠でその範囲をプレビュー表示すること]。
3. When [トリミングモーダルが画像を読み込んで初期表示するとき], the [Auth Profile UI] shall [画像の短辺がトリミング枠にぴったり収まる最小ズームを初期状態として設定し、枠内に不要な余白が生じないようにすること]。

**確定・キャンセル**
4. When [ユーザーがトリミング確定操作を行ったとき], the [Auth Profile UI] shall [選択範囲を正方形に切り出した画像を生成し、一辺512ピクセルを超える場合は比率を保ったまま一辺512ピクセル以下に縮小し、JPEG形式に変換した上でアバター領域にプレビュー表示し、モーダルを閉じること]。
5. When [ユーザーがトリミングモーダルでキャンセル操作を行ったとき], the [Auth Profile UI] shall [画像の選択および変更を一切反映せずにモーダルを閉じ、変更前のアバター表示を維持すること]。
6. If [トリミング処理中に画像の切り出しに失敗したとき], the [Auth Profile UI] shall [トリミングに失敗した旨の日本語エラーメッセージを表示し、変更前のアバター表示を維持すること]。
