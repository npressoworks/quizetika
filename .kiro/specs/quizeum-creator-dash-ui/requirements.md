# 要件定義書: quizetika-creator-dash-ui

## はじめに
本ドキュメントは、クイズ投稿SNS「quizetika」におけるクイズ作成・編集、クリエイターダッシュボード（アナリティクスおよび間違い指摘管理）、およびリストの作成・編集・エクスポートを含む、クリエイター（クリエイター）向けフロントエンドUI要件を定義します。

**Phase 6（2026-06）**: クイズエディタのジャンルセレクトを `metadata_genres` マスタ（`listActiveGenres`）駆動に切り替え、ジャンル新設可決後の選択肢反映と play-flow 探索 UI との ID 一貫性を確保する。

**Phase 8（2026-06）**: リスト作成時の `listType`（クイズリスト／問題リスト）選択、問題リスト編集（問題検索・アタッチ・DnD・エクスポート）、および作問エディタでの過去自作クイズ検索・参照リンク追加 UI を追加する（永続化・検証は `quizetika-core` が担当）。

**Phase 12（2026-06）**: クリエイターダッシュボード（`/creator/dashboard`）の Server Component 化、静的フレーム先行表示、および Suspense と Streaming による非同期スケルトン表示への移行を追加します。また、作問エディタの主要テキストエリア（説明文・問題文・真相・解説文）の自動伸長、過去自作クイズ検索の問題文・正解テキスト対応、参照リンク操作時の成功フィードバック、問題文ヒット時のアコーディオン自動展開、リンク済み問題の「リンク解除」、およびテストプレイ後の編集画面復帰時における問題数の不変保持を追加します（問題文・正解テキスト照合ロジックは `quizetika-core` が担当）。

**Phase 13（2026-06）**: 難易度の5段階化（1〜10から1〜5への変更）に伴い、作問エディタの難易度スライダー入力を1〜5の5段階へ変更します。

**Phase 20（2026-06-09）**: 〇✕問題（`true-false`）を出題形式として選択可能にし、作問時は正解トグル（「〇が正解」「×が正解」）のみで設定できる UI を追加します。選択肢テキストの自由編集は提供しません（永続化・検証は `quizetika-core` が担当）。

**Phase 26（2026-06-10）**: クイズリスト・問題リストの作成・編集・詳細・エクスポート UI を完全廃止します。クリエイターダッシュボードの「リスト作成」CTA および `/list/*` ルートを除去します（要件 26 参照）。要件 3・4・6 および要件 10 のリスト関連部分は本フェーズで **廃止** します。

**Phase 27（2026-06-28）**: クリエイターダッシュボードをプレイヤー統計情報も確認できる統合ダッシュボードへと拡張し、タブ（「プレイヤー」「クリエイター」）で表示を切り替え可能にする。プレイヤーダッシュボードには、基本プレイ統計、直近7日間のプレイ回数推移、プレイモード割合、よくプレイするジャンル／タグ、および正答率の高いジャンル／タグを表示する。

**Phase 28（2026-06-28）**: クリエイターダッシュボードの間違い指摘キューにおいて、各指摘を解決（解消）済みとするアクションを追加し、Firestoreの指摘ステータス更新（`resolveReport` 呼び出し）とダッシュボード表示の即時反映を連携する。

**Phase 40（2026-07-12）**: クリエイターダッシュボード（分析専念）とは別に、作成者が自分のクイズ全体を一覧・検索・管理するための専用画面（`/creator/quizzes`）を追加する。一覧・検索（キーワード／統合ステータス／ジャンル／タグ）、新規作成・編集への導線、公開・限定公開・非公開の切り替え、クイズ単位の未解決指摘件数バッジ表示を提供する。既存の `visibility`（`public`/`followers`/`private`）と Pro プラン制限（`quizetika-core` の `assertCanSetQuizVisibilitySync` 等）を初めて UI に露出する。あわせて、クリエイターダッシュボードの簡易クイズ一覧（`creator-quiz-list`）を本画面への導線に置き換える（要件 15〜18 参照）。

**Phase 41（2026-07-13）**: `quizetika-core` Phase 41 の tier リネームに伴い、要件 17 の「Pro プラン制限」表記を「Creator プラン制限」に改める。新設される中間価格帯プラン「Player」はクイズ限定公開・非公開への切り替え権利を持たず、Free と同様に非活性表示・有料プランへの案内導線の対象とする。

**Phase 42（2026-07-17）**: プレイヤーダッシュボードに、プレイ傾向を直感的に俯瞰できるワードクラウドセクションを追加する。プレイ済みクイズの「タグ」とクイズタイトル由来の「キーワード」を切り替え表示でき、語の大きさはプレイ回数、語の色は正答率を表現する（要件 20 参照）。

**Phase 44（2026-07-18）**: 統合ダッシュボード（`/creator/dashboard`）を、期間・ジャンル・タグ・問題形式・プレイモード等で絞り込めるフィルタバーと、集計値から履歴・設問別明細へ掘り下げるドリルダウンを備えたリッチダッシュボードへ刷新する（要件 21〜25 参照）。プレイヤー統計の集計母集団を「直近100件のクライアント集計」から `quizetika-core` が提供する全期間フィルタ付き集計（core 要件 37〜41）へ移行し、要件 13.7・13.8・20.4 の集計母集団規定を置き換える。クリエイタータブには期間内 KPI（ユニークプレイヤー数・完走率を含む）、トレンドグラフ、並び替え可能なクイズ別ランキング、およびクイズ単体分析ビュー（設問別解答分布・スコア分布・離脱分析）を追加する。フォロワー数はダッシュボードに表示しない。指摘 0 件時のモックデータ表示は撤去する。

  - 自分が作成したクイズデータのワンクリック一括 JSON エクスポート機能。
  - **Phase 8（リスト関連は Phase 26 で廃止）**: クイズエディタ内の過去自作クイズ検索パネルと問題の参照リンク追加 UI。

## 要件

### 要件 1: クイズ作成・編集画面 (Page: `/quiz/create`, `/quiz/[id]/edit`)
**目的:** クイズ作成者として、タグ正規化・サジェスト・厳格な検証付きのビジュアルエディタでクイズを下書き・編集・公開したい。それにより高品質なクイズを効率よく制作できる。

#### 受け入れ基準
1. クイズエディタ画面は、タイトル、説明、サムネイルアップロード、難易度（1〜5スライダー）、ジャンルセレクト、タグ入力（最大5個）のメタデータフィールドを表示すること。**Phase 6 以降**: ジャンルセレクトの選択肢は `listActiveGenres` 由来とする（要件 5 参照）。
2. クイズエディタ画面は、ジャンルセレクトの隣に「新しいジャンルを申請する」リンクを表示し、ジャンル新設申請画面へ遷移できること。
3. 作成者がタグを入力したとき、クイズエディタ画面は、テキストを正規化（トリム、小文字化、空白・記号除去）し、類似する canonical タグが存在する場合は「推奨: 類似するタグ #TagName が既に存在します。既存のタグを使用することをお勧めします。」のインライン警告を表示すること。
4. クイズエディタ画面は、問題の動的追加・削除、および問題タイプの「選択式 (multiple-choice)」／「短答文字入力式 (text-input)」切り替えを可能にすること。
5. 作成者がクイズの公開を試みたとき、クイズエディタ画面は、Zod スキーマ検証を実行し、統合制約違反（問題なし、正解未指定等）がある場合は明確な赤色エラーリストを表示すること。
6. クイズエディタ画面は、作成者が任意のタイミングで「下書き保存」をクリックし、検証エラーなく `QuizService.saveQuiz(status='draft')` 経由で下書きを Firestore に保存できること。

### 要件 2: クリエイターダッシュボード (Page: `/creator/dashboard`)
**目的:** クイズ作成者として、アナリティクス・指摘対応・エクスポートを一元的に行いたい。それによりパフォーマンスを把握しクイズを維持できる。

#### 受け入れ基準
1. クリエイターダッシュボード画面は、累計プレイ数・平均評価・ブックマーク数のアナリティクス概要を折れ線／棒グラフ等で表示すること。
2. クリエイターダッシュボード画面は、各クイズの問題別解答割合（選択肢の円グラフ等）を詳細パネルで表示すること。
3. クリエイターダッシュボード画面は、プレイヤーからのクローズド指摘（誤字、事実誤認、別解等）のキューを一覧表示すること。
4. 作成者が指摘カードの「修正する」をクリックしたとき、システムは、該当問題がプリロードされたクイズエディタ画面へ遷移すること。
5. クリエイターダッシュボード画面は、「クイズ一括エクスポート」ボタンで全クイズ（下書き含む）を単一のダウンロード可能な JSON パッケージにまとめること。
6. **Phase 28**: When 作成者が指摘カードの「解決済みにする」ボタンをクリックしたとき、the Creator Dashboard UI shall 指摘IDを解決済みとするAPI（`resolveReport`）を呼び出し、成功後に該当指摘をキュー一覧から即座に除外すること。
7. **Phase 28**: While 指摘の解決処理が実行中である間、the Creator Dashboard UI shall 該当指摘カード内の「解決済みにする」ボタンおよび「修正する」ボタンを非活性化（disabled）にし、二重送信を防止すること。

### 要件 3: クイズリスト詳細画面 (Page: `/list/[id]`) — **Phase 26 で全体廃止**
**目的:** Quizetika のユーザーとして、リストの内容を確認し連続プレイを開始したい。それにより整理されたクイズコレクションに取り組める。

#### 受け入れ基準
1. リスト詳細画面は、リストタイトル、説明、作成者アバター、カバー画像、およびクイズリストと問題リストを区別する種別表示（`listType` 未設定時はクイズリストとして表示）を表示すること。
2. リストがクイズリスト（`listType` が `quiz` または未設定）であるとき、リスト詳細画面は、収録クイズのカード一覧を表示すること。
3. リストが問題リスト（`listType` が `question`）であるとき、リスト詳細画面は、収録問題を順序付き一覧で表示し、問題文抜粋と親クイズタイトルを含めること（連続プレイ開始は `quizetika-play-flow-ui` 要件 11 が担当）。
4. ユーザーがクイズリストで「リストプレイ開始」をクリックしたとき、リスト詳細画面は、従来どおり `attempts.listId` と `mode: 'list'` によるクイズ単位の連続プレイを開始すること。
5. 認証済みユーザーがリスト作成者である場合、リスト詳細画面は、`listType` に関わらず「リストを編集する」ボタンを表示すること。

### 要件 4: リスト作成・編集画面 (Page: `/list/create`, `/list/[id]/edit`) — **Phase 26 で全体廃止**
**目的:** クイズ作成者として、検索とドラッグ＆ドロップでリストを編集したい。それによりテーマ別のコンテンツコレクションをパッケージ化できる。

#### 受け入れ基準
1. リスト編集画面は、タイトル、説明、カバー画像、公開／非公開トグル、および `listType` に応じたコンテンツアタッチパネル（要件 6）を表示すること。
2. リストタイプが `quiz` のとき、リスト編集画面は、クイズ検索・アタッチ、ドラッグ＆ドロップ並び替え、クイズリスト JSON エクスポートを従来どおり提供すること。
3. リストタイプが `question` のとき、リスト編集画面は、要件 6 に従い問題のアタッチ・並び替え・エクスポートを提供すること。

### 要件 5: クイズエディタのジャンルマスタ連携（Phase 6）
**目的:** クイズ作成者として、承認済みジャンルマスタに沿ったジャンルセレクトを使いたい。それにより探索・モデレーションルールと一致したジャンルで公開できる。

#### 受け入れ基準
1. クイズエディタ画面は、マウント時に `listActiveGenres`（または同等のコア API）で有効ジャンル一覧を取得し、各 option にマスタの `displayName` と値 `genreId`（ドキュメント ID）を表示すること。
2. クイズエディタ画面は、選択可能ジャンルの正本としてハードコードされた `<option>` 一覧を用いてはならない。
3. クイズエディタ画面は、ジャンルセレクトの隣に「新しいジャンルを申請する」リンクを維持し、`/community/genres` へ遷移できること。
4. 作成者がジャンル新設フローから戻ったあとエディタがフォーカスを得たとき、または明示的な再試行が行われたとき、クイズエディタ画面は、有効ジャンル一覧を再取得し、新規承認ジャンルをフルリロードなしで選択肢に反映すること。
5. 保存済み下書きの `genre` が現行の有効一覧に含まれないとき（レガシーまたはマージ保留）、クイズエディタ画面は、現在値を表示したまま保存を許可すること（公開時検証は `quizetika-core` に委譲）。
6. ジャンルマスタ取得に失敗した場合、クイズエディタ画面は、エラーまたは再試行 UI を表示し、サイレントにハードコード一覧へフォールバックしてはならない。
7. クイズエディタ画面は、選択した `genreId` を `QuizService.saveQuiz` / 公開フローへ従来どおりの形状で送信すること（公開時の canonical 解決は core が担当）。

### 要件 6: 問題リストの作成・編集（Phase 8）(Pages: `/list/create`, `/list/[id]/edit`) — **Phase 26 で全体廃止**
**目的:** クイズ作成者として、公開済み問題を複数ソースから集めて問題リストを作成・編集したい。それによりクイズ単位を超えた学習用コレクションを編集できる。

#### 受け入れ基準
1. 作成者がリスト編集画面で新規リストの作成を開始したとき、クリエイターダッシュボード UI は、保存前にリストタイプ `quiz` または `question` の選択を必須とし、`createQuizList` 経由で `listType` を永続化すること。
2. 既存リストを編集するとき、クリエイターダッシュボード UI は、現在の `listType` を表示し、作成後の `listType` 変更を許可してはならない。
3. リストタイプが `question` のとき、リスト編集画面は、クイズアタッチパネルを非表示にし、問題アタッチパネルを表示すること。
4. 作成者がアタッチする問題を検索したとき、リスト編集画面は、少なくとも次のソースでキーワード検索をサポートすること：自作公開クイズ内の問題、ブックマーク済み問題（`getBookmarkedQuestions`）、コアルールで追加可能な他者の公開問題。
5. 作成者が問題をアタッチ選択したとき、リスト編集画面は、`addQuestionToList` を呼び出し、問題文抜粋と親クイズタイトルをアタッチ一覧に表示すること。
6. 候補問題の親クイズが公開済みでない場合、作成者が当該問題のアタッチを試みたとき、クリエイターダッシュボード UI は、エラーを表示し、問題を追加してはならない。
7. 作成者がアタッチ済み問題を削除したとき、リスト編集画面は、`removeQuestionFromList` を呼び出し、フルページリロードなしで UI を更新すること。
8. 作成者がアタッチ済み問題の順序を変更したとき、リスト編集画面は、`reorderQuestionList`（または同等のコア API）を呼び出し、ドラッグ＆ドロップハンドルで順序を保持すること。
9. 作成者が問題リストのエクスポートを実行したとき、リスト編集画面は、`exportQuestionList` を呼び出し、コアのエクスポート規則に従ったリストメタデータと問題参照を含む JSON パッケージをダウンロードすること。
10. リストタイプが `quiz` のとき、リスト編集画面は、従来どおりのクイズ検索・アタッチ・並び替え・エクスポートを維持し、問題アタッチパネルを表示してはならない。

### 要件 7: 作問エディタの過去自作クイズ検索・参照リンク（Phase 8）(Pages: `/quiz/create`, `/quiz/[id]/edit`)
**目的:** クイズ作成者として、過去の自作クイズから問題を検索し、参照リンクで新規クイズに再利用したい。それにより再入力なしで効率的に作問できる。

#### 受け入れ基準
1. クイズエディタ画面は、認証済み作成者向けに折りたたみ可能なパネル「過去のクイズから問題を追加」（または同等のラベル）を表示すること。
2. 作成者がパネルでキーワードを入力またはタグを選択したとき、クイズエディタ画面は、現在の作成者にスコープした `searchAuthorQuizzes` を呼び出し、下書きを含む一致クイズを一覧表示すること。
3. 作成者が検索結果のクイズを展開したとき、クイズエディタ画面は、当該クイズの問題を問題文抜粋付きで表示し、個別問題のリンク選択を可能にすること。
4. 作成者が自作過去クイズ内の問題をリンク確認したとき、クイズエディタ画面は、エディタ状態に参照リンク（`linkKind: 'reference'`）として追加し、UI 層で問題内容を複製しないこと。
5. 作成者が所有しないクイズの問題をリンクしようとした場合、クリエイターダッシュボード UI は、当該操作を提供しない、または試行時に禁止エラーを表示すること。
6. クイズエディタ画面は、参照リンク済み問題と新規作成問題を視覚的に区別すること（例：バッジ「参照リンク」）。
7. 作成者が参照リンク済み問題の内容を編集したとき、クイズエディタ画面は、下書き保存または公開保存の前に、コアの Copy-on-Write 方針に従い保存時に切り離し用のローカルコピーが作成され得る旨を作成者に通知すること。
8. 作成者が下書きから参照リンク済み問題を削除したとき、クイズエディタ画面は、エディタ上からのみ参照を解除し、元クイズの問題レコードを削除してはならない。
9. クイズエディタ画面は、参照リンク済み問題 ID を `QuizService.saveQuiz` / 公開フローへ送信し、内容が未変更の場合は core が問題ドキュメントを複製せず参照として永続化できること。
10. クリエイターダッシュボード UI は、参照の永続化、所有権チェック、重複ドキュメント防止をクライアントで実装してはならない（いずれも `quizetika-core` の責務）。
11. When 作成者が過去自作クイズ検索パネルでキーワードを入力したとき、クイズエディタ画面は、クイズのタイトルおよび説明文に加え、各問題の問題文および問題タイプに応じた正解テキスト（選択式・正誤式の正解選択肢文、記述式・即答式・連想式の正解候補、並び替え式の各要素文、ウミガメのスープの必須正解キーワード）でも一致する自作クイズを検索結果に含めること。
12. The クイズエディタ画面は、過去自作クイズ検索パネルのキーワード入力欄に、タイトル・説明・問題文・正解テキストが検索対象であることを示すプレースホルダーまたは説明文を表示すること。
13. When 作成者が過去自作クイズ検索パネルで問題のリンク操作を成功したとき、クイズエディタ画面は、リンクが完了したことを作成者が認識できる成功メッセージを表示すること。
14. If 作成者が既にリンク済みの問題を検索パネルから操作するとき、クイズエディタ画面は、「リンク」ボタンの代わりに「リンク解除」ボタンを表示し、リンク解除によりエディタ上の当該問題のみを削除すること（要件 7.8）。重複「リンク」操作は防止すること。
15. When 作成者が過去自作クイズ検索パネルでキーワードを入力し、いずれかの問題の問題文がキーワードに一致したとき、クイズエディタ画面は、当該クイズのアコーディオンを自動展開し、一致する問題を即時表示すること。問題文ヒットの問題は正解テキストのみヒットの問題より先頭に表示してよい。
16. When 作成者がリンク済み問題の「リンク解除」を実行したとき、クイズエディタ画面は、リンク解除が完了したことを作成者が認識できるフィードバック（成功メッセージ等）を表示してよい。

### 要件 8: 作問エディタのテキストエリア自動伸長（Phase 12）(Pages: `/quiz/create`, `/quiz/[id]/edit`)
**目的:** クイズ作成者として、長文を入力するときにテキストエリアが内容に合わせて自動的に広がるようにしたい。それによりスクロールや手動リサイズなしで快適に作問できる。

#### 受け入れ基準
1. The クイズエディタ画面は、クイズ全体の説明文、各問題の問題文、ウミガメのスープ問題の真相入力、各問題の解説文のテキストエリアを表示すること。
2. When 作成者が上記テキストエリアに複数行のテキストを入力または削除したとき、クイズエディタ画面は、入力内容の行数に応じてテキストエリアの表示高さを自動的に拡張または縮小すること。
3. While テキストエリアが自動伸長しているとき、クイズエディタ画面は、作成者による手動リサイズも許可してよい。
4. The クイズエディタ画面は、タイトル・タグ・必須正解キーワード入力等の単一行用途フィールドを、本要件の自動伸長対象に含めてはならない。
5. When 作成者が既存の下書きまたは編集対象クイズを開いたとき、クイズエディタ画面は、保存済みの長文が含まれる上記テキストエリアについて、初回表示時点で内容に見合った高さを表示すること。

### 要件 9: テストプレイと編集画面復帰（Phase 12）(Pages: `/quiz/create`, `/quiz/[id]/edit`, `/quiz/test-play/*`)
**目的:** クイズ作成者として、テストプレイ後に編集途中のドラフトを失わず編集画面へ戻りたい。それにより本番保存前にプレイ検証と作問の継続を安全に行える。

#### 受け入れ基準
1. 作成者がクイズエディタからテストプレイを開始したとき、クイズエディタ画面は、未保存のエディタ状態（問題配列を含む）を `sessionStorage` に保存し、テストプレイ画面へ遷移すること。
2. 作成者がテストプレイ画面または結果画面から「編集画面に戻る」を選択したとき、システムは、元の編集パス（`/quiz/create` または `/quiz/[id]/edit`）へ `fromTestPlay=1` クエリ付きで遷移すること。
3. When 編集画面が `fromTestPlay=1` でマウントされたとき、クイズエディタ画面は、認証ユーザーと `sourcePath` が一致する `sessionStorage` のドラフトを1回だけ消費し、エディタ状態へ復元すること。
4. When テストプレイドラフトの復元に成功したとき、クイズエディタ画面は、復元した問題数を変更せず表示し、Firestore からの通常ロード（`getQuiz`）および新規作成時のデフォルト問題追加を実行してはならない。
5. When テストプレイドラフトの復元が完了したとき、クイズエディタ画面は、`fromTestPlay` クエリを除去した URL へ置換遷移し、同一セッション内の再マウントで問題が重複しないこと。
6. If `fromTestPlay=1` が付与されているが認証ユーザーが未確定のとき、クイズエディタ画面は、通常ロード（`getQuiz` / デフォルト問題追加）を開始してはならない。
7. If テストプレイ payload が欠落または期限切れのとき、クイズエディタ画面は、クエリを除去して復帰し、問題の重複追加を行ってはならない（編集モードでは以降の通常ロードに委ねてよい）。

### 要件 10: ダッシュボードの非同期表示最適化（Phase 12 追加）
**目的:** クイズ作成者として、クリエイターダッシュボードにアクセスした際、画面全体の白紙ローディングを待つことなく、静的なヘッダーや新規作成アクションが即座に表示され、データが揃った箇所から順番にダッシュボードの分析結果や指摘キューが表示されるようにしたい。これにより、待機時のストレスや画面の点滅による不快感を防ぎ、即時のアクション（クイズ作成等）を開始できる。

#### 受け入れ基準

**クリエイターダッシュボード画面における非同期表示最適化**
1. When [クイズ作成者がクリエイターダッシュボード（`/creator/dashboard`）にアクセスしたとき], the [Creator Dashboard UI] shall [サーバーコンポーネントとしてヘッダー枠、タイトル、および「クイズ新規作成」等のアクションボタンを含む静的なページフレームを即座にレンダリングし、Next.jsのStreaming機能を通じてクライアントへ送信すること]。
2. While [累計統計データ（プレイ数、評価、ブックマーク数等）がロード中である間], the [Creator Dashboard UI] shall [統計カードの表示エリアに、専用のスケルトンプレースホルダーを表示すること]。
3. When [累計統計データのロードが完了したとき], the [Creator Dashboard UI] shall [スケルトン表示領域を、実際の統計カードデータ（数値等）に差し替えること]。
4. While [作成クイズ一覧データがロード中である間], the [Creator Dashboard UI] shall [クイズ一覧表示エリアに、専用のローディングスケルトンを表示すること]。
5. When [作成クイズ一覧データのロードが完了したとき], the [Creator Dashboard UI] shall [スケルトン表示領域を、実際のクイズ行リストに差し替えること]。
6. While [間違い指摘フィードバックキューがロード中である間], the [Creator Dashboard UI] shall [指摘一覧表示エリアに、専用のローディングスケルトンを表示すること]。
7. When [間違い指摘フィードバックキューのロードが完了したとき], the [Creator Dashboard UI] shall [スケルトン表示領域を、実際の指摘カード一覧に差し替えること]。
8. While [アナリティクスグラフデータ（プレイ数、評価推移）がロード中である間], the [Creator Dashboard UI] shall [グラフ表示エリアに、専用のグラフ用スケルトンを表示すること]。
9. When [アナリティクスグラフデータのロードが完了したとき], the [Creator Dashboard UI] shall [スケルトン表示領域を、実際のチャート（折れ線・円グラフ等）に差し替えること]。

**アクセシビリティ・テスト支援**
10. The [Creator Dashboard UI] shall [統計カードのスケルトン領域に `data-testid="stats-skeleton"`、クイズ一覧のスケルトン領域に `data-testid="quiz-list-skeleton"` を付与すること]。
11. The [Creator Dashboard UI] shall [指摘キューのスケルトン領域に `data-testid="feedback-list-skeleton"`、グラフのスケルトン領域に `data-testid="charts-skeleton"` を付与すること]。

**クイズ作成・編集画面、リスト関連画面の非同期表示最適化**
12. When [ユーザーがクイズ作成画面（`/quiz/create`）または編集画面（`/quiz/[id]/edit`）にアクセスしたとき], the [Creator Dashboard UI] shall [サーバーコンポーネントとして戻るボタン、入力フォームの静的フレーム（見出し、保存・公開等のアクションエリア）を即座にレンダリングし、Next.jsのStreaming機能を通じてクライアントへ送信すること]。
13. While [編集対象のクイズデータやジャンルマスタ、タグリストの取得中である間], the [Creator Dashboard UI] shall [フォーム内部の各エリア（メタデータ入力、問題アコーディオン等）に専用のスケルトンプレースホルダーを表示すること]。
14. When [データのロードが完了したとき], the [Creator Dashboard UI] shall [スケルトン表示領域を、実際の編集フォームコンテンツに差し替えること]。
15. When [ユーザーがリスト詳細画面（`/list/[id]`）またはリスト作成・編集画面（`/list/create`, `/list/[id]/edit`）にアクセスしたとき], the [Creator Dashboard UI] shall [サーバーコンポーネントとして戻るボタン、コンテナ等の静的フレームを即座にレンダリングし、クライアントへ送信すること]。
16. While [該当リストデータやアタッチ対象のクイズ・問題データがロード中である間], the [Creator Dashboard UI] shall [リストコンテンツ表示領域に専用のスケルトンを表示すること]。

### 要件 17: 公開・限定公開・非公開の統合ステータス表示と切り替え（Phase 40）(Page: `/creator/quizzes`)
**目的:** クイズ作成者として、クイズごとの現在の公開範囲を一目で把握し、その場で公開範囲を切り替えたい。それにより下書き・限定共有・一般公開を状況に応じて使い分けられる。

#### 受け入れ基準
1. The Creator Quiz Management UI shall 各クイズの状態を、下書き（`status: 'draft'`）・公開（`status: 'published'` かつ `visibility: 'public'`）・限定公開（`status: 'published'` かつ `visibility: 'followers'`）・非公開（`status: 'published'` かつ `visibility: 'private'`）・審査により非表示（`status: 'suspended'`）のいずれか1つの統合ステータスとして表示すること。
2. Where クイズが下書き（`status: 'draft'`）である場合、the Creator Quiz Management UI shall 公開範囲切り替え操作を表示せず、編集画面（要件 1）での公開操作へ委ねること。
3. Where クイズがモデレーションにより凍結（`status: 'suspended'`）である場合、the Creator Quiz Management UI shall 下書きとは区別される統合ステータス表示（例:「審査により非表示」）とし、公開範囲切り替え操作を表示しないこと。
4. Where クイズが公開済み（`status: 'published'`）である場合、the Creator Quiz Management UI shall 公開・限定公開・非公開の3値から選択できる切り替え操作を提供すること。
5. When 作成者が公開範囲の切り替え操作を行ったとき、the Creator Quiz Management UI shall 選択した公開範囲を反映し、成功時は一覧上の統合ステータス表示を即時更新すること。
6. If 公開範囲の切り替えに失敗した場合、the Creator Quiz Management UI shall エラーメッセージを表示し、切り替え前の公開範囲表示を維持すること。
7. Where 作成者が Creator プラン（またはそれ以上）の権利を保有していない場合（Free および Player を含む）、the Creator Quiz Management UI shall 限定公開・非公開への切り替え操作を非活性状態で表示し、Creator プランへの案内導線を提示すること。
8. If 有料プランの権利を保有しない作成者が限定公開・非公開への切り替えを試みてサーバー側の権限エラーを受け取った場合、the Creator Quiz Management UI shall エラーメッセージと有料プランへの案内導線を表示し、切り替え前の公開範囲表示を維持すること。
9. The Creator Quiz Management UI shall 統合ステータス表示に `data-testid` プレフィックス（例: `creator-quiz-status-draft` / `creator-quiz-status-public` / `creator-quiz-status-followers` / `creator-quiz-status-private` / `creator-quiz-status-suspended`）を付与すること。
10. The Creator Quiz Management UI shall 公開範囲切り替え操作領域に `data-testid="creator-quiz-visibility-toggle"` を付与すること。

### 要件 18: クイズ単位の未解決指摘件数表示（Phase 40）(Page: `/creator/quizzes`)
**目的:** クイズ作成者として、どのクイズに未対応の指摘が溜まっているかを一覧から把握したい。それにより対応が必要なクイズを見落とさず編集画面へ移動できる。

### 要件 19: クリエイターダッシュボードとの統合（Phase 40）(Page: `/creator/dashboard`)
**目的:** クイズ作成者として、ダッシュボードでは分析情報に集中しつつ、クイズ管理が必要なときはワンタップで管理画面に移動したい。それにより2画面の役割が重複せず迷わず使い分けられる。

2. The Creator Dashboard UI shall 要件 2・要件 13 に定義された分析機能（アナリティクスグラフ、個別問題解答割合、プレイヤー統計等）を本統合による変更対象に含めないこと。
3. When 作成者がクリエイターダッシュボードから当該導線を選択したとき、the Creator Dashboard UI shall `/creator/quizzes` へ遷移すること。
4. The Creator Dashboard UI shall 当該導線に `data-testid="creator-dashboard-manage-quizzes-link"` を付与すること。

### 要件 20: プレイヤーワードクラウド（Phase 42）(Page: `/creator/dashboard`)
**目的:** プレイヤーとして、自分がよく遊んでいるテーマと、その中での得意・苦手を1つのビジュアルで直感的に把握したい。既存のジャンル・タグ分析（要件 13）は上位5件のリスト表示に留まるため、より広い範囲のプレイ傾向を一望できるワードクラウドが欲しい。

#### 受け入れ基準

**表示と切り替え**
1. The [Dashboard UI] shall プレイヤータブ内にワードクラウドセクションを表示し、当該セクションに `data-testid="player-word-cloud"` を付与すること。
2. The [Dashboard UI] shall ワードクラウドの表示対象として「タグ」と「キーワード」の2つを切り替えられる操作を提供し、初期表示は「タグ」とすること。
3. When 表示対象を切り替えたとき、the [Dashboard UI] shall ページ遷移や再読み込みなしに、選択した対象のワードクラウドへ表示を差し替えること。

**集計ロジック**
4. ~~The [Dashboard UI] shall 要件 13.7 と同一の集計元データ（直近最大 100 件の完了済みプレイ履歴）を用いてワードクラウドを構成すること。~~ — **Phase 44 で改定**（集計母集団は要件 22.7 のフィルタ適用後集計に従う）
5. When 「タグ」表示のとき、the [Dashboard UI] shall プレイ済みクイズに付与されたタグごとにプレイ回数を集計し、プレイ回数の多い上位最大 30 語を表示すること。
6. When 「キーワード」表示のとき、the [Dashboard UI] shall プレイ済みクイズのタイトルから抽出したキーワードごとにプレイ回数を集計し、プレイ回数の多い上位最大 30 語を表示すること。
7. The [Dashboard UI] shall キーワード抽出において、助詞・記号・数字のみの語、および「クイズ」「問題」「検定」等のクイズタイトルに頻出する汎用語を表示対象から除外すること。
8. When 同一クイズのタグまたはタイトル内に同じ語が複数回含まれるとき、the [Dashboard UI] shall 1回のプレイにつき当該語のプレイ回数を1回として集計すること。

**視覚表現**
9. The [Dashboard UI] shall 各語の文字サイズを、プレイ回数が多い語ほど相対的に大きく表示すること。
10. The [Dashboard UI] shall プレイ回数が3回以上の語について正答率（正解数 / 総出題数）を算出し、正答率の高低を語の色の区分として表現すること（対象閾値は要件 13.6 と同一）。
11. Where 語のプレイ回数が3回未満である場合、the [Dashboard UI] shall 当該語を「データ不足」を示す控えめな色区分で表示すること。
12. The [Dashboard UI] shall 色区分の意味（正答率の高低およびデータ不足）を示す凡例をセクション内に表示すること。
13. When ユーザーが語をポイントまたはフォーカスしたとき、the [Dashboard UI] shall 当該語のプレイ回数と正答率（プレイ回数3回未満の場合はデータ不足である旨）を確認できる手段を提供すること。
14. The [Dashboard UI] shall 同一の集計データに対して、再表示のたびに語の並びが変わらない一貫した配置で表示すること。

**空状態**
15. If 表示対象の語が1件も存在しない場合、the [Dashboard UI] shall ワードクラウドの代わりに「データがありません」等の空状態メッセージを表示すること。

### 要件 21: ダッシュボード分析フィルタバー（Phase 44）(Page: `/creator/dashboard`)
**目的:** ユーザーとして、ダッシュボードの統計を期間や属性で絞り込み、見たい範囲の分析に集中したい。それにより自分の関心に沿った傾向把握ができる。

#### 受け入れ基準
1. The [Dashboard UI] shall プレイヤータブに、期間（直近7日・直近30日・直近90日・全期間）、ジャンル、タグ、問題形式、プレイモードのフィルタバー（`data-testid="dashboard-filter-bar"`）を表示すること。
2. The [Dashboard UI] shall クリエイタータブに、期間（同上の4区分）、ジャンル、出題形式、公開状態のフィルタバーを表示すること。
3. The [Dashboard UI] shall 期間フィルタの初期値を「直近30日」、その他のフィルタの初期値を「すべて」とすること。
4. When ユーザーがフィルタ条件を変更したとき、the [Dashboard UI] shall ページ遷移や再読み込みなしに、当該タブ内の各統計セクションをフィルタ適用後の集計結果へ更新すること。
5. While フィルタ適用後の集計結果を取得中である間、the [Dashboard UI] shall 該当セクションにローディング表示（スケルトンまたは同等）を表示し、直前の結果と混在した表示を行わないこと。
6. The [Dashboard UI] shall 適用中のフィルタ条件を画面上で確認できるように表示し、すべてのフィルタを初期値へ戻す一括リセット操作を提供すること。
7. When ユーザーがプレイヤー／クリエイタータブを切り替えたとき、the [Dashboard UI] shall 各タブのフィルタ状態を独立に保持し、相互に影響させないこと。
8. If 集計結果の取得に失敗した場合、the [Dashboard UI] shall エラーが発生した旨の表示と再試行の手段を提供すること。

### 要件 22: プレイヤーダッシュボードの全期間フィルタ集計刷新（Phase 44）(Page: `/creator/dashboard`)
**目的:** プレイヤーとして、直近100件に限定されない全プレイ履歴に基づく統計を、フィルタで絞り込みながら確認したい。それにより長期的なプレイ傾向と得意・苦手を正確に把握できる。

#### 受け入れ基準
1. The [Dashboard UI] shall プレイヤー統計をコアの全期間フィルタ付き集計（`quizetika-core` 要件 37 の契約）から取得して表示し、直近最大100件のクライアント側集計を行わないこと（要件 13.7 の置換）。
2. The [Dashboard UI] shall 基本統計カードに、既存4指標（累計プレイ数・平均正解率・平均解答時間・ユニーククイズ数）に加えて、合計プレイ時間および連続プレイ日数（ストリーク）を表示すること。
3. The [Dashboard UI] shall プレイ回数・正答率の推移グラフを、フィルタ期間に応じた粒度（直近7日・30日は日別、直近90日は週別、全期間は月別）で表示すること（要件 13.8 の推移部分の置換）。
4. The [Dashboard UI] shall ジャンル別・タグ別・プレイモード別に加えて、設問形式別のプレイ実績（解答問題数と正答率）の内訳セクション（`data-testid="player-format-analysis"`）を表示すること。
5. The [Dashboard UI] shall 正答率の高いジャンル・タグ（既存）に加えて、正答率の低いジャンル・タグ（母数3件以上、昇順、最大5件）を「苦手分析」として表示すること。
6. When ユーザーが苦手分析のジャンル・タグ項目から探索導線を選択したとき、the [Dashboard UI] shall 当該ジャンル・タグのクイズを探せる画面への導線を提供すること（プレイ開始後の画面は担当外）。
7. The [Dashboard UI] shall ワードクラウド（要件 20）の集計母集団を、適用中のフィルタ条件に連動させること（要件 20.4 の置換）。
8. The [Dashboard UI] shall 既存の `data-testid`（`player-stats`、`player-charts`、`player-genre-tag-analysis`、`player-word-cloud`）を維持すること。
9. If フィルタ適用後の母集団に試行が1件も存在しない場合、the [Dashboard UI] shall 各セクションに空状態メッセージを表示すること。

### 要件 23: プレイヤードリルダウン（Phase 44）(Page: `/creator/dashboard`)
**目的:** プレイヤーとして、集計値から該当するプレイ履歴、さらに1回のプレイの設問別の正誤内容まで掘り下げたい。それにより具体的な間違いを振り返って学習に活かせる。

#### 受け入れ基準
1. When ユーザーが内訳・分析セクションの項目（ジャンル・タグ・設問形式・プレイモード）を選択したとき、the [Dashboard UI] shall 適用中のフィルタ条件と当該項目の条件を合成した該当プレイ履歴一覧（`data-testid="player-drilldown"`）を表示すること。
2. The [Dashboard UI] shall ドリルダウン履歴一覧をページング（初回一定件数＋続きの読み込み）付きで表示すること。
3. When ユーザーが履歴一覧のエントリを選択したとき、the [Dashboard UI] shall 当該試行の設問ごとの正誤・自分の解答内容・解答時間・ヒント使用数を表示すること。
4. If 設問別明細が保存されていない過去の試行が選択された場合、the [Dashboard UI] shall 試行全体の要約のみを表示し、設問別明細が存在しない旨の注記を表示すること。
5. The [Dashboard UI] shall ドリルダウン表示から元の集計表示へ戻る導線を提供すること。

### 要件 24: クリエイターダッシュボードの期間集計刷新（Phase 44）(Page: `/creator/dashboard`)
**目的:** クリエイターとして、自作クイズ群への反響を期間で絞り込んで把握し、伸びているクイズ・属性を比較したい。それにより次の作問改善に活かせる。

#### 受け入れ基準
1. The [Dashboard UI] shall クリエイター統計をコアのフィルタ付き集計（`quizetika-core` 要件 40 の契約）から取得し、期間内プレイ回数・ユニークプレイヤー数・獲得ブックマーク数・獲得評価数と平均評価・完走率を統計カードとして表示すること。
2. The [Dashboard UI] shall プレイ回数・ブックマーク獲得・評価獲得の推移グラフを、フィルタ期間に応じた粒度（要件 22.3 と同一規則）で表示すること。
3. The [Dashboard UI] shall 自作クイズごとの期間内プレイ回数・平均正答率・獲得ブックマーク数・獲得評価を一覧するクイズ別ランキング（`data-testid="creator-quiz-ranking"`）を表示し、いずれの指標でも並び替えできること。
4. When ユーザーがクイズ別ランキングの行を選択したとき、the [Dashboard UI] shall 当該クイズのクイズ単体分析ビュー（要件 25）へ遷移すること。
5. The [Dashboard UI] shall 自作クイズに対する設問形式別の解答数と正答率のパフォーマンスセクションを表示すること。
6. If 完走率の算出対象データ（試行ライフサイクル記録の導入以降の試行）が不足している場合、the [Dashboard UI] shall 完走率の値の代わりまたは併記で、データが蓄積中である旨の注記を表示すること。
7. The [Dashboard UI] shall 間違い指摘キュー（既存 Phase 28 の解決アクションを含む）の表示を維持すること。
8. If 未解決の指摘が0件の場合、the [Dashboard UI] shall 空状態メッセージを表示し、モック（ダミー）の指摘データを表示しないこと（既存実装のモック注入の撤去）。
9. If フィルタ適用後の母集団に自作クイズまたは試行が存在しない場合、the [Dashboard UI] shall 各セクションに空状態メッセージを表示すること。

### 要件 25: クイズ単体分析ビュー（Phase 44）(Page: `/creator/dashboard`)
**目的:** クリエイターとして、特定の自作クイズの設問ごとの正答状況・解答分布・スコア分布・離脱状況を確認したい。それにより問題単位の難易度調整や改善点を特定できる。

#### 受け入れ基準
1. When ユーザーがクイズ別ランキング（要件 24.4）からクイズを選択したとき、the [Dashboard UI] shall 当該クイズのクイズ単体分析ビュー（`data-testid="creator-quiz-analysis"`）を表示すること。
2. The [Dashboard UI] shall 設問ごとの累計正答率、および選択式・〇✕設問の選択肢別解答分布（累計）を表示し、これらが期間フィルタの対象外（累計値）である旨を明示すること。
3. The [Dashboard UI] shall 適用中の期間フィルタに基づく完了試行の得点帯別スコア分布を表示すること。
4. The [Dashboard UI] shall 未完了試行が最後に到達した設問位置の分布と完走率を離脱分析として表示すること。
5. If 離脱分析の算出対象データが不足している場合、the [Dashboard UI] shall データが蓄積中である旨の注記を表示すること。
6. The [Dashboard UI] shall クイズ単体分析ビューからクリエイターダッシュボードへ戻る導線を提供すること。
7. If 他ユーザーが作成したクイズの分析データが取得できない場合（不正な遷移等）、the [Dashboard UI] shall エラーまたは空状態を表示し、内容を表示しないこと。
