# Requirements Document

## Introduction
本ドキュメントは、Firebase Auth から Supabase Auth への認証システム完全移行における要件を定義します。対象となるのは、マルチプロバイダ認証、クライアントサイドの認証状態管理、Next.js ミドルウェアによるセッション更新・アクセス制限、および API エンドポイントにおけるトークン検証です。

## Boundary Context
- **In scope**:
  - 各種認証プロバイダ（Google, Twitter/X, Microsoft, メール/パスワード）によるログイン・サインアップ・ログアウトフロー。
  - クライアントサイドの認証状態同期およびデータベース上の初期プロフィール作成。
  - ミドルウェアでの認証セッションCookie更新およびルート保護（BANユーザー、未認証、権限不足の制御）。
  - API エンドポイントにおけるリクエストトークンの検証と UID 特定。
- **Out of scope**:
  - Firebase Auth から既存ユーザーデータを Supabase Auth へエクスポート・インポートする作業。
  - データベース操作自体の Firebase から Supabase への移行（`supabase-core-data` が担当）。
  - UI デザイン・レイアウトの変更。
- **Adjacent expectations**:
  - `supabase-foundation` 仕様で定義された Supabase クライアント初期化ロジック（`client.ts`, `server.ts`, `middleware.ts`）を利用すること。

## Requirements

### 1. 認証プロバイダ連携機能
**Objective:** ユーザーとして、多様なログイン手段を利用して安全にアカウントを作成・認証したい。

#### Acceptance Criteria
1. When ユーザーが Google、Twitter/X、または Microsoft のログインボタンをクリックした時、認証モジュールは各 OAuth プロバイダを介した外部認証フローを開始する。
2. When ユーザーがメールアドレスとパスワードを入力して送信した時、認証モジュールは入力情報に基づきメールアドレス認証によるログインまたは新規登録を実行する。
3. When ユーザーがログアウト操作を行った時、認証モジュールは現在のセッションを破棄し、サインアウト状態に移行する。

### 2. 認証状態の管理とプロフィール同期
**Objective:** ユーザーとして、ログインした状態が各ページで維持され、プロフィール情報が適切に認識されるようにしたい。

#### Acceptance Criteria
1. While ユーザーがログイン状態である時、認証コンテキストはセッションから一意のユーザー識別子（UID）およびメールアドレス等の基本情報を保持する。
2. When 認証状態が変化した（ログインまたはログアウトした）時、システムはクライアント Cookie 内の UID 等 of 認証関連 Cookie を同期または削除する。
3. When 新規サインアップ直後などでデータベース上に該当 UID のプロフィールが存在しない時、システムはデータベースに初期値（デフォルト表示名、アバター、自己紹介）を設定したプロフィールレコードを作成する。

### 3. ミドルウェアセッション更新とルートガード
**Objective:** 管理者および開発者として、未認証・BANユーザー・権限不足ユーザーによる特定ルートへのアクセスを防ぎたい。

#### Acceptance Criteria
1. When ユーザーがページにアクセスした時、ミドルウェアは認証 Cookie の有効期限を自動的に更新（セッションローテーション）する。
2. If ユーザーの BAN 状態を示す Cookie（`quizetika_banned=true`）が検出された場合、ミドルウェアは `/banned` 以外のすべての保護ルートへのアクセスを遮断し、BAN 告知ページ（`/banned`）へ強制リダイレクトする。
3. If 未認証ユーザーがログイン必須ルート（`/bookmarks`, `/creator/dashboard`, クイズ編集等）にアクセスした場合、ミドルウェアはアクセスを拒否し、ログイン画面へリダイレクトする。
4. If 管理者権限（`role = admin`）を持たないユーザーが管理画面（`/admin/*`）にアクセスした場合、ミドルウェアは 404 (not-found) ページへリダイレクトする。

### 4. API エンドポイントのトークン検証
**Objective:** 開発者として、API エンドポイントで受信したリクエストが正当なユーザーから送信されたものであることを検証したい。

#### Acceptance Criteria
1. When API エンドポイントが認証を必要とするリクエストを受信した時、システムは Authorization ヘッダーに含まれる Bearer トークンを検証し、正当なユーザーの UID を特定する。
2. If トークンが無効、改ざんされている、または期限切れである場合、システムは HTTP 401 Unauthorized エラーを返却してリクエストを却下する。
